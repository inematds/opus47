🔎 Pesquisar codebase legado

Goal: mapear o módulo de billing legado em src/legacy/billing/.

Paths: src/legacy/billing/**/*.py

Return shape:
  1. Entrypoints públicos (função + assinatura + arquivo:linha)
  2. Dependências externas (imports não-stdlib)
  3. Estado compartilhado (globais, singletons, db sessions)
  4. Riscos conhecidos (TODOs, FIXMEs, hacks)

Não modifique nada. Apenas leia e resuma.
2

✅ Verificar contra spec

Goal: comparar implementação em src/feature/x.ts com docs/spec-x.md.

Return shape (tabela markdown):
  | Requisito | Implementado? | Evidência (arquivo:linha) | Gap |

Prioridade: coverage completa, não ranking. Liste TODOS
os requisitos mesmo que parciais.
3

📚 Docs de git diff

Goal: gerar entry de CHANGELOG.md para commits desde
v1.2.0 (git log v1.2.0..HEAD).

Return shape (markdown):
  ### [Unreleased]
  #### Added / Changed / Fixed / Removed
  - <rationale em 1 linha focando em "why">

Não inclua commits de merge nem de docs/. Agrupe por tema.
4

🧪 Suite de testes pesada

Goal: rodar `pytest -v tests/integration/` e reportar.

Return shape (tabela):
  | Teste | Status | Arquivo:linha | Tipo de erro |

Não inclua stdout. Se >50 falhas, agrupe por tipo e
mostre 1 exemplo por grupo. Termine com nota sobre
flakiness suspeita.
5

🎨 Design alternativo

Dispare 3 subagentes no mesmo turno, cada um com constraints diferentes:

Sub A

Design minimalista, <200 linhas, sem deps novas.

Sub B

Design event-driven, usando biblioteca X existente.

Sub C

Design performático, otimiza para 10k req/s.

6

🔍 Security review

System: You are a security auditor. Assume adversarial input.

Goal: revisar diff `git diff main..HEAD` em busca de
vulnerabilidades.

Report every finding — coverage > ranking.

Return shape (tabela):
  | Issue | Arquivo:linha | Severity | Confidence | Fix sugerido |

Severity: critical/high/medium/low. Confidence: 1–5.
